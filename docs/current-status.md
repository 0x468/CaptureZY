# 当前阶段总览

## 文档目的

这份文档用于同步 CaptureZY 截至当前提交阶段的实际进展，并作为下一阶段规划讨论的基础草案。

它不替代 `docs/roadmap.md`，而是补充“当前已经做到什么”“还缺什么”“下一步先做什么”。

## 当前结论

- 截图主链路已经打通，不再只是原型。
- 贴图主链路已经具备基础可用性。
- 设置基础设施已经起步，Alpha 可用性收敛已进入执行阶段。
- 下一阶段重点应从“继续补单点功能”切换到“收敛交互、配置与稳定性边界”。

## 当前能力清单

### 已具备的截图能力

- 区域截图
- 全屏截图
- 将截图复制到剪贴板
- 将截图直接保存为 PNG
- 截图后自动创建贴图窗口

### 已具备的贴图能力

- 创建独立贴图窗口
- 无边框置顶显示
- 拖动贴图窗口
- 滚轮缩放
- 缩放百分比临时浮框
- 右键菜单基础操作
- 复制与保存入口
- 批量显示全部贴图
- 批量隐藏全部贴图
- 批量关闭全部贴图

### 已具备的平台能力

- 托盘驻留
- 全局热键触发截图
- 配置文件自动生成
- 配置文件编辑入口、目录入口与重载入口
- 托盘内快速切换默认截图范围与默认截图动作
- 基础主窗口状态展示
- COM 初始化与资源释放

### 已具备的工程能力

- CMake 工程组织
- Win32 原生模块化结构
- `clang-format`
- `clang-tidy`
- 基础产品、架构、决策与路线图文档

## 当前模块状态

### `core`

- 负责应用状态、状态文案与设置模型。
- 当前已包含配置对象与 JSON 持久化入口，边界仍相对清晰。

### `feature_capture`

- 已从“直接操作剪贴板的抓屏函数”演进为“截图结果对象 + 多种输出路径”。
- 当前已包含覆盖层、抓屏、保存对话框、PNG 导出和截图结果对象。

### `feature_pin`

- 已包含贴图窗口与贴图管理器。
- 当前已具备基础可用交互，但离最终产品级体验还有明显距离。

### `platform_win`

- 当前是主要调度入口。
- 负责托盘、热键、截图请求、截图结果分发和贴图批量控制。
- 目前功能集中度较高，后续仍需要持续控制复杂度。

### `app`

- 负责进程入口和 COM 生命周期。
- 当前职责比较稳定，没有明显膨胀。

## 已完成的关键里程碑

1. 工程骨架、格式化和静态检查配置完成。
2. Win32 应用壳层、托盘和全局热键完成。
3. 区域截图覆盖层与选区链路完成。
4. 剪贴板复制链路完成。
5. 贴图窗口原型完成，并逐步补齐移动、缩放、关闭和反馈浮层。
6. `CaptureResult` 与 `PinManager` 已引入，截图与贴图的对象边界更稳定。
7. PNG 保存流程与全屏截图命令已接入主路径。
8. 配置存储、默认截图行为、默认保存目录、热键配置与托盘配置入口已接入。

## 当前主要问题

### 交互仍然偏“功能已到位”，但还不够“产品化”

- 托盘菜单、贴图右键菜单和快捷键之间还缺少更统一的语义。
- 截图后的默认行为目前仍更像“开发期固定策略”，而不是可配置产品行为。

### 持久化层已起步，但还没有图形化设置界面

- 热键、默认截图动作、默认保存路径与基础命名前缀已经可配置。
- 当前已经具备托盘中的快速切换入口，但主要配置入口仍是 JSON 配置文件，不是图形化设置界面。
- 贴图状态和设置尚未持久化。

### 稳定性边界还需要专门处理

- 高 DPI 细节仍需要系统性验证。
- 多显示器与显示器热插拔行为还没有形成回归清单。
- 大尺寸贴图缩放的渲染和性能仍需继续观察。

### 代码组织还有继续收敛空间

- `platform_win/main_window.*` 仍是当前最容易继续膨胀的位置。
- 后续如果命令继续增多，需要考虑进一步拆出命令编排或应用控制层。

## 下一阶段建议目标

建议将下一阶段定义为“Alpha 可用性收敛”。

该阶段不追求大而全，而是优先让已有主路径更稳定、更一致、更像真正可日用的产品。

## 建议优先级

### 第一优先级：设置与默认行为

- 增加设置存储结构
- 支持可配置热键
- 支持默认截图动作配置
- 支持默认保存路径与基础命名规则

这是下一阶段最值得优先做的部分，因为它直接决定现有能力能否真正形成稳定工作流。

### 第二优先级：贴图交互收敛

- 统一贴图右键菜单项
- 明确复制、保存、关闭、隐藏、置顶等操作语义
- 视需要补充轻量 Hover 操作入口
- 评估透明度调节是否进入 Alpha 范围

### 第三优先级：稳定性与回归基线

- 建立 DPI 手工验证清单
- 建立多显示器验证清单
- 建立截图保存与贴图窗口基础回归清单
- 明确发布前需要人工验证的关键场景

### 第四优先级：渲染层升级

- 评估贴图窗口是否从当前 GDI 路径逐步切到更稳定的 D2D 路径
- 评估覆盖层是否需要更现代的视觉表达

这一项建议后置，因为它会带来额外技术复杂度，但对当前“可用性收敛”不是最高优先级。

## 建议的短期任务拆分

### 里程碑 A：设置基础设施

- 已完成：定义配置对象与 JSON 存储格式
- 已完成：增加配置读写入口
- 已完成：接入默认截图行为
- 已完成：接入默认保存目录
- 已完成：接入热键配置
- 已完成：托盘中的配置文件编辑、目录打开与重载入口
- 已完成：托盘中的默认截图范围与默认截图动作快速切换
- 未完成：图形化设置界面

### 里程碑 B：贴图交互收敛

- 收敛贴图右键菜单
- 明确批量操作与单窗口操作边界
- 评估是否增加“恢复所有贴图”的入口提示
- 梳理贴图关闭与隐藏的状态一致性

### 里程碑 C：验证与发布前准备

- 形成手工回归清单
- 补齐崩溃诊断或最小日志策略
- 收敛 README 与路线图中的状态描述

## 暂不建议优先投入的方向

- OCR
- 复杂标注工具
- 云同步
- 安装器与自动更新
- 过早做大规模渲染重构

这些方向并非不重要，而是与当前阶段的“把现有能力做稳、做顺手”相比优先级更低。

## 当前阶段的完成判定建议

当满足以下条件时，可以认为项目进入了更明确的 Alpha 阶段：

- 截图默认行为可以配置
- 基础热键可以配置
- PNG 保存路径与命名规则具备最小可用性
- 贴图窗口单体与批量操作语义清晰
- DPI、多显示器、保存与贴图流程具备基础回归清单
