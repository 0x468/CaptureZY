# 当前阶段总览

## 文档目的

这份文档用于同步 CaptureZY 截至当前提交阶段的实际进展，并作为下一阶段规划讨论的基础草案。

它不替代 `docs/roadmap.md`，而是补充“当前已经做到什么”“还缺什么”“下一步先做什么”。

## 当前结论

- 截图主链路已经打通，不再只是原型。
- 贴图主链路已经具备基础可用性。
- 设置基础设施和基础持久化能力已经基本站稳。
- 当前阶段更接近“Beta 打磨起步”，重点应从“补基础能力”切换到“收敛交互细节、智能选区边界与代码组织”。
- 截图后的基础调整态外壳已经成型，已支持二次确认、边角缩放、反向翻折、选区移动与尺寸反馈。
- 当前最值得优先推进的主线已经从“先把调整态做出来”切换为“补截图后工具条最小闭环，再继续收敛智能选区边界质量”。
- 控件级识别暂不建议直接进入正式功能开发，优先保留为 feasibility 调研方向。

## 当前模块状态

### `core`

- 负责应用状态、状态文案与设置模型。
- 当前已包含配置对象、JSON 持久化、日志与崩溃诊断相关基础能力，边界仍相对清晰。

### `feature_capture`

- 已从“直接操作剪贴板的抓屏函数”演进为“截图结果对象 + 多种输出路径”。
- 当前已包含覆盖层、抓屏、窗口级预框选、保存对话框、PNG 导出和截图结果对象。

### `feature_pin`

- 已包含贴图窗口与贴图管理器。
- 当前已具备基础可用交互，但离最终产品级体验还有明显距离。

### `platform_win`

- 当前是主要调度入口。
- 负责托盘、热键、截图请求、截图结果分发和贴图批量控制。
- 主窗口当前仅作为隐藏宿主窗口存在，不再承担可见状态面板职责。
- `main_window` 已开始按流程拆分，但平台调度职责仍然偏集中，后续仍需要持续控制复杂度。

### `app`

- 负责进程入口、COM 生命周期与顶层崩溃诊断安装。
- 当前职责仍比较稳定，但已经开始承担最小运行时兜底责任。

## 已完成的关键里程碑

1. 工程骨架、格式化和静态检查配置完成。
2. Win32 应用壳层、托盘和全局热键完成。
3. 区域截图覆盖层与选区链路完成。
4. 剪贴板复制链路完成。
5. 贴图窗口原型完成，并逐步补齐移动、缩放、关闭和反馈浮层。
6. `CaptureResult` 与 `PinManager` 已引入，截图与贴图的对象边界更稳定。
7. PNG 保存流程与全屏截图命令已接入主路径。
8. 配置存储、默认截图行为、默认保存目录、热键配置、托盘点击动作与退出确认已接入。
9. `main_window` 已按 capture / settings / tray 流程完成首轮拆分。

## 当前主要问题

### 交互仍然偏“功能已到位”，但还不够“产品化”

- 贴图的更多进阶交互仍未收敛，例如透明度、悬停入口与更细的状态反馈。
- 截图覆盖层已具备窗口级自动预框选、自由拖拽、快捷全屏选择与基础调整态，但窗口可见边界、系统壳窗口处理以及截图后工具条闭环仍未完全收敛。
- 截图交互设置已补齐热键、默认动作、托盘点击动作和退出确认，但独立的截图设置页与更细的交互语义仍未收敛。

### 持久化层和设置界面已具备基础可用性，但范围还没有完全收敛

- 热键、默认截图动作、托盘单击/双击动作、默认保存路径与基础命名前缀已经支持统一的图形化编辑。
- 退出确认与“不再提示”已经支持持久化。
- 当前保留了托盘中的默认范围/动作快速切换，完整配置则收敛到统一设置对话框。
- 贴图状态、更多截图交互设置和更细的产品级偏好尚未持久化。

### 稳定性边界还需要专门处理

- 已具备最小崩溃日志、dump 输出和基础运行日志框架，关键链路也已具备会话级关联能力。
- 已对当前尺寸下的贴图缩放结果增加位图缓存，但大尺寸贴图的渲染和性能仍需继续观察。
- 贴图拖拽已切换为自定义移动逻辑，以避免系统标题栏拖拽带来的顶部自动贴边行为。
- 已显式启用进程级 Per-Monitor DPI 感知，但各窗口在不同 DPI 组合下仍需要持续回归。
- 已形成多显示器与热插拔基础观察项，但更细的回归矩阵和发布前验证目前暂不作为第一优先级。
- 隐藏贴图的恢复提示当前主要依赖托盘菜单语义，后续如需更强反馈可再考虑轻量提示方案。

### 代码组织还有继续收敛空间

- `platform_win/main_window.*` 已开始拆分，但 `platform_win` 仍是当前最容易继续膨胀的位置。
- 后续如果命令继续增多，需要考虑进一步拆出命令编排或应用控制层。

## 下一阶段建议目标

建议将下一阶段定义为“Beta 打磨起步”。

该阶段不追求大而全，而是优先让已有主路径更顺手、更一致，并逐步补齐智能选区与交互语义上的短板。

## 建议优先级

### 第一优先级：截图后编辑壳收敛

- 增加截图后的最小工具条闭环，例如确认、贴图、复制、保存、取消
- 收敛截图后 `Enter`、左键单击、右键、`Esc` 与后续工具条按钮之间的交互语义
- 保持当前基础调整态的无闪烁体验，不因为补工具条而退回整层重绘

这是当前最值得优先推进的部分，因为截图后的基础调整态已经具备最小可用性，继续补齐工具条闭环可以最快把它推进到更接近产品化的状态。

### 第二优先级：智能选区增强

- 修正窗口预选与实际可见边界的一致性
- 增加任务栏与系统壳窗口识别
- 保持窗口预选与自由拖拽、自定义全屏选择之间的交互一致性
- 后续再评估控件级预框选与 UI Automation 集成

这仍然是截图主链路中的关键质量项，但在当前阶段应排在“工具条最小闭环”之后推进。

### 第三优先级：贴图交互收敛

- 继续打磨复制、保存、关闭、隐藏、置顶等操作的反馈细节
- 优先处理滚轮缩放时的闪烁、跳变与不丝滑问题
- 视需要补充轻量 Hover 操作入口
- 评估透明度调节是否进入当前阶段范围
- 评估隐藏贴图恢复反馈是否需要更明确的轻量提示

### 第四优先级：截图交互设置收敛

- 在设置对话框中补出独立的截图设置页
- 继续收敛截图页中的默认行为、取消语义和托盘交互语义
- 评估是否还需要补充更细的截图中提示或状态反馈

### 第五优先级：代码组织收敛

- 继续控制 `platform_win` 的职责膨胀
- 视命令增长情况进一步拆分主流程编排逻辑
- 保持当前模块边界，不为了未来假设过度抽象

### 第六优先级：稳定性与回归基线

- 继续观察大尺寸贴图缩放与渲染性能
- 在真实环境中持续执行 DPI / 多显示器 / 热插拔观察项
- 逐步补充更细的回归矩阵，但这一项暂不抢占前四项优先级

### 第七优先级：发布准备

- 保留基础 ZIP 打包与安装目录导出能力
- 安装器、签名与发布自动化继续后置
- 在核心交互进一步收敛前，不优先投入发布包装工作

## 建议的短期任务拆分

2. 再收敛截图后的交互语义，例如 `Enter`、左键单击、右键、`Esc` 与工具条按钮之间的关系。

4. 控件级识别暂不直接进入正式功能开发，只先做 feasibility 调研，确认 UI Automation 与常见应用类型的命中质量。

## 暂不建议优先投入的方向

- OCR
- 复杂标注工具
- 云同步
- 安装器与自动更新
- 过早做大规模渲染重构

这些方向并非不重要，而是与当前阶段的“把现有能力做稳、做顺手”相比优先级更低。

## 当前阶段的完成判定建议

当满足以下条件时，可以认为项目进入了更明确的 Beta 打磨阶段：

- 智能选区的窗口边界与系统壳窗口行为基本稳定
- 截图设置页与主要默认行为语义完成收敛
- 贴图窗口的常用交互反馈足够清晰
- `platform_win` 的主流程复杂度没有继续失控膨胀
- DPI、多显示器与热插拔观察项具备可重复执行的基础验证路径
