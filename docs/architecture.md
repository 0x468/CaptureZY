# 架构说明

## 当前方向

CaptureZY 采用 Win32 原生桌面应用路线，并明确区分业务逻辑、Windows 集成层与渲染层。

## 规划中的模块边界

- `core`
  - 应用命令与状态
  - 与具体功能无关的业务规则
  - 不直接依赖 Win32
- `platform_win`
  - Win32、COM、Shell、剪贴板、热键、DPI 与显示器集成
  - 原生窗口生命周期辅助能力
  - 托盘图标与驻留式应用行为
- `render_d2d`
  - Direct2D 与 DirectWrite 渲染
  - 位图、画刷与文字资源管理
- `feature_capture`
  - 区域截图流程
  - 覆盖层窗口协调
  - 截图管线编排
- `feature_pin`
  - 贴图模型与交互
  - 贴图窗口生命周期与持久化挂钩
- `app`
  - 进程启动
  - 依赖组装
  - 顶层消息循环

## 当前入口能力

- 托盘菜单已作为主入口，可触发区域截图、全屏截图、保存类截图、设置入口和贴图批量操作
- `Ctrl+Shift+A` 作为默认全局热键触发截图入口，热键已支持图形化配置
- 主窗口当前仅作为隐藏宿主窗口存在，主要负责托盘、热键、截图请求分发和贴图库存摘要
- 区域截图入口已接入覆盖层、窗口级预框选、拖拽框选、`Ctrl+A` 全屏和 `Esc` 取消
- 截图结果已统一进入复制、保存或贴图分发链路

## 依赖规则

- `core` 必须保持平台无关。
- Windows API 不应进入 `core`。
- 功能模块可以依赖 `core`、`platform_win` 与 `render_d2d`。
- `app` 负责组装模块，但不应堆积业务逻辑。

## 线程方向

初期实现保持保守：

- 一个 UI 线程拥有原生窗口和消息循环
- 只有在截图、编码或持久化确有需要时再引入后台线程
- 在剖析证明有必要之前，渲染流程避免无意义的线程切换

## 渲染方向

- 使用 `Direct2D` 与 `DirectWrite` 绘制二维图形和文字
- 截图覆盖层与贴图窗口使用原生 Win32 窗口
- 渲染状态尽量贴近窗口实例，避免形成大范围可变全局状态

## 当前实现状态

- `core` 已承载应用状态、设置模型、JSON 配置读写和崩溃诊断。
- `feature_capture` 已承载抓屏、覆盖层、截图结果对象、默认保存与 PNG 对话框保存。
- `feature_pin` 已承载贴图窗口、缩放缓存、右键菜单和批量贴图管理。
- `platform_win` 已成为当前主要调度层，集中处理托盘、热键、设置对话框与截图结果分发。
- `app` 负责进程入口、COM 生命周期、DPI 感知初始化与顶层异常兜底。
- `render_d2d` 目前仍基本处于占位状态，当前实际渲染主路径仍以 GDI 为主，后续再评估是否逐步替换。

## 当前风险

- `platform_win/main_window.*` 已经承担较多命令编排职责，后续需要继续控制复杂度。
- 截图与贴图主链路已经成型，但 DPI、多显示器、热插拔和性能回归仍主要依赖手工验证。
