# 架构说明

## 当前方向

CaptureZY 采用 Win32 原生桌面应用路线，并明确区分业务逻辑、Windows 集成层与渲染层。

## 规划中的模块边界

- `core`
  - 应用命令与状态
  - 与具体功能无关的业务规则
  - 不直接依赖 Win32
- `platform_win`
  - Win32、COM、Shell、剪贴板、热键、DPI 与显示器集成
  - 原生窗口生命周期辅助能力
  - 托盘图标与驻留式应用行为
- `render_d2d`
  - Direct2D 与 DirectWrite 渲染
  - 位图、画刷与文字资源管理
- `feature_capture`
  - 区域截图流程
  - 覆盖层窗口协调
  - 截图管线编排
- `feature_pin`
  - 贴图模型与交互
  - 贴图窗口生命周期与持久化挂钩
- `app`
  - 进程启动
  - 依赖组装
  - 顶层消息循环

## 当前入口能力

- 托盘菜单可触发“开始截图”
- `Ctrl+Shift+A` 作为全局热键触发截图入口
- 当前截图入口先切换应用状态并更新主窗口展示，后续再接入覆盖层与选区流程
- 当前已接入最小覆盖层原型：全屏遮罩、左键占位完成、`Esc` 取消

## 依赖规则

- `core` 必须保持平台无关。
- Windows API 不应进入 `core`。
- 功能模块可以依赖 `core`、`platform_win` 与 `render_d2d`。
- `app` 负责组装模块，但不应堆积业务逻辑。

## 线程方向

初期实现保持保守：

- 一个 UI 线程拥有原生窗口和消息循环
- 只有在截图、编码或持久化确有需要时再引入后台线程
- 在剖析证明有必要之前，渲染流程避免无意义的线程切换

## 渲染方向

- 使用 `Direct2D` 与 `DirectWrite` 绘制二维图形和文字
- 截图覆盖层与贴图窗口使用原生 Win32 窗口
- 渲染状态尽量贴近窗口实例，避免形成大范围可变全局状态

## 说明

当前源码树已经包含 `core`、`platform_win`、`render_d2d` 与 `app` 的最小骨架，但这些模块目前仍以启动流程和边界约束为主。后续应随着截图、贴图与托盘功能落地，再继续向各模块填充真实实现。
